# 账号与权限

更新时间：2026-07-14

## 当前模型

- 后台用户角色只有 `admin` 与 `user`，状态只有 `active` 与 `disabled`。
- 管理员拥有全部地图功能权限，并管理账号、系统设置、地图创建/归档、运行数据清理和全局审计。
- 普通用户只能看到 `map_permissions` 中已授予 `map.view` 的地图；每个具体功能还需对应的地图权限。
- 游戏客户端不创建后台会话，只使用独立 API Key；公开群抽参与者不属于后台用户。

权限完整枚举由 `server/middleware/auth.js` 生成到 `.ai/generated/code-facts.md`。

## 会话流程

1. 登录校验用户名、密码、用户状态和限流。
2. 服务端生成不可预测 Token，只将哈希写入 `sessions`，Cookie 返回浏览器。
3. `loadSession` 通过 Token 哈希、有效期和用户状态加载身份。
4. 退出、密码修改、管理员重置密码或停用账号时删除相关会话。

## 授权规则

- `requireAuth` 只确认有效登录；`requireAdmin` 再确认管理员角色。
- `requireMapPermission` 解析正整数地图 ID；管理员直接获得全部权限，普通用户必须同时拥有 `map.view` 和目标权限。
- 管理员页面的前端 `AdminOnly` 只负责导航体验，服务端 `/api/admin` 仍统一执行 `requireAuth + requireAdmin`。
- 地图授权更新自动补入 `map.view`；空权限数组会删除该用户的地图授权记录。

## 审计与验证

- 登录、退出、资料修改、密码修改、账号创建/更新/重置、地图权限和系统设置变化均写入审计日志。
- 验证重点：停用与重置后的旧会话立即失效；普通用户不能越权访问未授权地图或管理接口；管理员能力不依赖前端显隐。

主要证据：`server/middleware/auth.js`、`server/routes/auth.js`、`server/routes/admin.js`、`src/auth/AuthContext.jsx`。
