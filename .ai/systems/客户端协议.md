# 游戏客户端协议

更新时间：2026-07-16

面向接入方的请求示例见 `docs/游戏客户端接入.md`；本文件只维护开发合约。

## 身份边界

- 游戏端专属接口统一挂载在 `/api/fq`，请求携带 `FQ-Map-Key`；服务端用哈希查找有效 API Key，并写入最近使用时间。
- Key 固定地图、环境和权限；完整 Token 只在创建响应中展示一次，数据库只保存哈希与前缀。
- 每个接口使用独立 `game.*` 权限，权限清单由源码生成到 `.ai/generated/code-facts.md`。

## 写入语义

- 玩家存档按 `map_id + environment + player_uid` 唯一保存；地图全局存档按 `map_id + environment` 唯一保存。
- 存档保存提交完整 JSON 快照、`expectedRevision` 和以 `FQ-` 开头的 `requestId`。版本匹配时 revision 加一；最近一次相同请求可安全重放，相同 ID 改写内容或旧版本覆盖均返回 409。
- 开局 `POST /api/fq/bootstrap` 最多批量读取 24 个玩家及当前环境全局存档；未创建记录返回 revision 0 和空对象。存档封禁玩家不下发已有快照且不能写入。
- 玩家上报按 `map_id + environment + uid` upsert，并刷新最后活跃时间。
- 指标按地图、环境和日期 upsert；日志对相同维度内容聚合次数；埋点只增加已启用点位。
- 排行榜批量上报最多 500 条，相同 UID 更新实时条目；榜单必须存在、启用且属于 Key 环境。
- 风险事件使用客户端 `eventId` 作为幂等键；相同地图、环境和事件键重试不重复创建。

## 可靠投递

- 消息和礼包遵循“拉取待处理记录 → 游戏内写入成功 → ACK”。
- 未 ACK 记录可以再次返回，客户端必须按消息 ID 或发放 ID 自身幂等。
- ACK 必须核对 UID、地图和环境，不能确认其他玩家或其他 Key 范围的记录。

## 响应与日志

- 成功返回 `{ success: true, data }`；失败返回 `{ success: false, error: { code, message }, requestId }`。
- 客户端可以记录 `requestId`、状态码和业务错误码，不得记录完整 Key、密码或玩家敏感资料。

主要证据：`server/routes/game.js`、`server/middleware/auth.js`、`server/routes/maps.js`、`server/db/migrations/004_fq_archives.sql`。
