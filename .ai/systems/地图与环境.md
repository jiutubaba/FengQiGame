# 地图与环境

更新时间：2026-07-22

## 地图生命周期

- 地图由管理员创建，保存名称、描述、默认环境和 `active`/`archived` 状态。
- 普通用户只能读取已授权地图；归档由管理员执行，保留关联数据并退出活跃运营范围。
- 永久删除与归档是独立操作，仅管理员可执行；前端连续两次确认风险，服务端仍要求请求中的地图 ID 和名称与锁定后的数据库记录完全一致。
- 永久删除在事务内写入 `map.delete` 审计并删除地图；外键级联清除所有环境的权限、玩家、存档、配置、榜单、风控、礼包资格、群抽、日志和 API Key。
- 上传目录先原子改名为受控 `.deleting-map-*` 暂存目录；数据库失败时恢复，成功后同时清理暂存目录和原地图目录。启动时只重试地图已不存在的受控残留目录。
- 删除后的历史审计保留且 `map_id` 置空；已有数据库备份、上传卷快照和 OSS 副本不受影响，仍按原保留期管理。
- 地图配置以 `map_configs` 保存 JSON 配置与预加载代码，更新需要 `map.edit` 并写审计。

## 环境隔离

- 运行环境固定为 `release`、`lobby`、`test`。
- 玩家、指标、日志、主播、埋点、API Key、消息、礼包资格、群抽、排行榜和风险事件等运行数据均按实际表结构使用 `map_id + environment` 隔离。
- API Key 创建时绑定地图和环境；客户端请求体无权覆盖二者。
- 前端环境切换必须重新请求目标环境，不能在客户端混合或复用另一环境缓存作为结果。

## 运行数据清理

- 只有管理员可执行，且请求必须明确确认目标地图名称。
- 当前事务删除目标环境的玩家消息、礼包资格、排行榜快照与实时条目、风险事件、玩家、日志和指标，并将埋点触发数归零。
- 当前事务保留地图、地图配置、礼物定义、排行榜定义、风控规则、主播、埋点定义、群抽活动、文件和 API Key；调整范围前必须审查外键和恢复策略。
- 操作必须写入 `map.runtime.clear` 审计记录。

## 验证

- 三个环境写入同 UID 后互不覆盖。
- 普通用户不能通过替换 URL 中 `mapId` 访问未授权地图。
- 归档、永久删除、清理和配置更新分别验证权限、事务、审计与刷新后持久化。
- 永久删除验证普通用户拒绝、确认信息不匹配、重复请求、数据库回滚后的目录恢复、所有关联表清空、API Key 失效、审计保留和磁盘无残留。

主要证据：`server/routes/maps.js`、`server/db/migrations/`、`src/pages/MapCenter.jsx`、`src/pages/MapWorkspace.jsx`。
