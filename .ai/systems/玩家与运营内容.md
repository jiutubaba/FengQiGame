# 玩家与运营内容

更新时间：2026-07-22

## 玩家

- 唯一边界为 `map_id + environment + uid`；名称不是唯一标识。
- 玩家包含等级、游戏等级、资料 JSON、最后活跃时间，以及物品、存档、榜单三类封禁。
- 后台新增、修改、删除需要玩家管理权限并写审计；客户端 upsert 只更新允许上报的资料和活跃时间。

## 消息与礼包

- 后台消息记录主题、内容、附件、创建人和送达状态；客户端成功写入游戏后 ACK 才标记送达。
- 礼包定义属于地图配置，默认数值为 `0`；`gift_entitlements` 以地图、环境、玩家、礼包为唯一当前资格，正数 upsert，设置 `0` 删除。
- `GET/PUT /api/maps/:mapId/gifts/entitlements` 使用 `gifts.manage` 权限，批量设置必须完整校验玩家环境和礼包地图归属并写审计。
- 游戏端 `/api/fq/deliveries/query` 每局返回 `{ gift_key, name, value }` 正数资格快照；不受 `dataBan` 影响，不写玩家存档、不累计、无礼包 ACK。消息仍保持写档成功后 ACK。
- 《沧澜》以现役福利礼包名称作为 `gift_key`；只有完全同名的正数资格会直接激活对应礼包，未知 Key 忽略。

## 主播、埋点、指标与日志

- 主播和埋点按地图与环境管理；相同名称或 Key 的唯一性由迁移约束决定。
- 指标按日期存储；日志按业务字段聚合并更新时间，删除需要授权和审计。
- 通用资源路由仍必须使用明确的权限常量，不能因为共享实现而放宽权限。

## 验证

- 封禁更新、消息、礼包、主播、埋点和日志管理均检查环境、权限、数据库结果和审计。
- 消息验证“未 ACK 可重取、ACK 后不重复”；礼包验证默认零值、批量覆盖、设零取消、重复查询不累计、地图/环境隔离、存档封禁不阻断、权限和审计。
- UI 空状态和筛选结果必须来自真实 API，不预置演示玩家。

主要证据：`server/routes/maps.js`、`server/routes/game.js`、`server/db/migrations/007_gift_entitlements.sql`。
