# 代码风格与组织

## 基本原则

- 命名直接表达职责，避免 `utils2`、`common-new`、`temp` 等泛名。
- 优先让一个业务流程在相邻代码中完整可读；只有确有复用或边界价值时抽取 helper。
- 保持现有 ESM、React 函数组件、Express Router、Zod 校验和参数化 SQL 风格。
- 不引入与需求无关的抽象、兼容层、回退分支或新依赖。
- 长文件修改使用小补丁，先定位真实调用链，避免覆盖同文件中的其他有效改动。

## 前端

- 页面不得用硬编码样本替代 API 数据；加载、空状态、错误和权限不足必须可区分。
- 权限菜单与按钮显隐复用服务端返回权限，但不能被视为安全校验。
- 保持现有深色运营工作台和金色强调体系；响应式断点修改必须检查桌面与窄屏。
- `MapWorkspace.jsx` 承载多业务域；新增复杂功能前先评估是否形成独立组件，禁止继续无边界堆叠。

## 服务端与数据库

- 所有外部输入先经 Zod 或等价明确校验；SQL 必须参数化。
- 多表原子操作使用 `transaction`；错误使用统一 `HttpError`/错误处理中间件。
- 新表或字段只通过递增迁移新增；已发布迁移不得原地改写。
- 变更权限、路由、环境变量或迁移后必须刷新自动事实。

## 文档与注释

- 注释解释业务原因、边界或危险点，不复述语句本身。
- 稳定合约写入对应 `systems/` 文档；长期取舍写 ADR；一次性过程只进入会话归档。
- `.ai/generated/` 只能由工具生成。
