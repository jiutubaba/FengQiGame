# 开发与运维工程边界

## 开发环境

- Docker 可用时按 Compose 运行完整拓扑；暂时不可用时可使用本机 PostgreSQL 原生模式。
- 原生模式只用于本机开发和内网验收，不提供 Caddy HTTPS，也不等同生产部署通过。
- `.env`、`.runtime/`、`uploads/`、`backups/`、数据库目录、日志和测试产物不得进入 Git。

## 数据库变更

- 迁移文件只追加，不修改已发布历史；应用启动时按文件名顺序应用未执行迁移。
- 运行迁移、恢复或清理前确认目标连接；正式恢复必须先做额外备份并停止应用写入。
- 数据库备份与上传卷是两套恢复对象，必须分别制定快照与演练策略。

## 发布与恢复

- 更新顺序：备份 → 更新代码 → 迁移/构建 → 健康检查 → 业务冒烟 → 保留回退证据。
- 正式公网只开放 80/443；PostgreSQL 不映射公网端口。
- 未在隔离环境恢复过的备份不能被声明为“可恢复”。
- 具体命令以 `docs/部署与运维.md` 和 `运维/` 脚本为准，本文件只维护工程约束。

## AI 文档维护

- 代码合约变化后先刷新自动事实，再更新稳定系统文档和会话归档。
- 部署事实只记录实际执行结果；Docker、证书或真实域名尚未验收时保留为 backlog。
- 远程 Git 使用私有 GitHub 仓库；`.github/workflows/ci.yml` 在 `main` 推送、Pull Request 和手工触发时运行 Node 24 质量检查、生产依赖审计及 PostgreSQL 17 隔离集成测试。
- GitHub 当前套餐不支持为该私有仓库启用 branch protection/rulesets；CI 成功不等于 `main` 已强制 required checks，不得把未启用的远程保护写成已完成。
