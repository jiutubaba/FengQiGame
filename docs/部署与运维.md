# 部署与运维

## 生产前提

- 一台可运行 Docker Compose 的 Windows 或 Linux 服务器。
- 正式域名 `fengqigame.com` 的注册局审核已通过，网站 A/AAAA 记录已指向服务器；企业邮箱 MX/SPF/CNAME 记录必须保留。
- 防火墙只对公网开放 80/443 业务端口；SSH 22 只允许管理员固定 `/32`，PostgreSQL 不映射宿主机端口。
- `.env`、数据库备份和上传文件不进入 Git，也不通过聊天或工单明文传输。

当前正式环境为广州 ECS `8.148.249.198`、Ubuntu 24.04、4 vCPU/8 GiB/100 GiB、5 Mbps。Caddy 已取得 `fengqigame.com` 的有效证书；ICP备案审核期间暂停网站 A 记录，公网健康接口暂不作为已通过项。备案前只通过服务器本机 `--resolve` 或 SSH 隧道验收，不提前恢复公网解析。

## 首次部署

Windows 运行 `首次配置.ps1`，Linux 可复制 `.env.example` 为 `.env` 并手工填写。正式环境必须满足：

- `SITE_ADDRESS=https://后台域名`
- `COOKIE_SECURE=true`
- `TRUST_PROXY=1`
- `ADMIN_PASSWORD` 至少 6 位且只用于本系统，不复用参考站密码
- `POSTGRES_PASSWORD` 与 `DATABASE_URL` 中的密码完全一致

启动：

```powershell
docker compose up -d --build
docker compose ps
docker compose logs --tail 200 app
```

应用启动会在数据库健康后按文件名顺序运行尚未执行的迁移，并且仅在数据库完全没有用户时创建初始管理员。

## 无法重启时的本机原生模式

Windows 固件虚拟化暂未开启、Docker Desktop 不能运行时，`启动网站.bat` 会自动选择 `native`。该模式要求本机已安装并运行 PostgreSQL 与 Node.js：

- 首次配置只临时读取 PostgreSQL 管理员密码，用于创建 `fengqi` 数据库和无超级用户权限的 `fengqi` 角色；该密码不会写入 `.env`。
- 初始管理员由配置脚本直接写入数据库，管理员明文密码不会保留在原生模式 `.env`。
- 应用只监听 `http://127.0.0.1:3000`，运行日志位于 `.runtime/`，上传文件仍位于 `uploads/`。
- `运维/备份数据.ps1` 和 `运维/恢复数据.ps1` 会按 `DEPLOYMENT_MODE` 自动选择本机 PostgreSQL 工具或容器流程；原生恢复同样会先安全停止应用，成功后重新启动并检查健康状态。

原生模式没有 Caddy 自动 HTTPS，不能直接作为公网发布方式。电脑可以重启后，应在 BIOS 开启 SVM 并切换到 Docker 模式完成最终生产验收。

## 更新

先备份，再拉取或覆盖新版本，然后执行：

```powershell
.\运维\备份数据.ps1
docker compose up -d --build
docker compose ps
```

不要删除 `postgres_data`、`uploads_data` 或 Caddy 数据卷。`docker compose down` 不删除数据；`docker compose down -v` 会删除全部命名卷，只允许在明确销毁环境时使用。

## 代码托管与 CI

远程仓库为公有 GitHub `origin`。`.github/workflows/ci.yml` 会在 `main` 推送、Pull Request 和手工触发时运行：

- Node 24 下的 `npm run check`；
- `npm run audit:prod`；
- PostgreSQL 17 本机隔离测试库上的 `npm run test:integration`。

`main` 已启用分支保护：上述质量/审计作业和隔离 PostgreSQL 集成测试均为 required checks，分支必须保持最新，管理员同样不能绕过；强推和删除 `main` 均被禁止。当前未强制 Pull Request 审批，单人维护者仍可在两项检查成功后合并。

## 备份与恢复

手工备份：

```powershell
.\运维\备份数据.ps1 -RetentionDays 14
.\运维\备份上传文件.ps1 -RetentionDays 14
```

两条本机脚本都支持 `-PassThru` 返回生成的非空备份文件，供正式异机上传编排使用。正式 Linux 服务器通过 `运维/备份到OSS.ps1` 顺序生成数据库和上传卷快照，以 `PutObject` 和 `--server-side-encryption AES256` 上传到独立私有 OSS 的 `prod/YYYY/MM/DD/`，并比较本地与远端 CRC64；任一步骤失败都会以非零状态退出并尝试发送邮件告警。

正式备份使用独立的服务器配置，不复用应用 `.env`，也不把真实值写入 Git：

```text
FENGQI_OSS_BUCKET=fengqigame-prod-backups-cn-wuhan-20260715
FENGQI_OSS_CONFIG=/etc/fengqigame/ossutilconfig
FENGQI_BACKUP_ALERT_TO=<告警收件邮箱>
FENGQI_MSMTP_CONFIG=/etc/msmtprc
```

- Bucket 固定为标准存储、武汉 LRS、私有 ACL、阻止公共访问和 SSE-OSS AES-256；`prod/` 对象 14 天后删除，未完成分片 1 天后删除。
- RAM 用户只绑定 `运维/阿里云OSS最小权限策略.json`，只允许列举、上传和读取校验该 Bucket 的 `prod/` 前缀，不授予删除和权限管理能力。
- 使用 ossutil 2.x 交互式写入 `/etc/fengqigame/ossutilconfig`，Endpoint 为 `https://oss-cn-wuhan-lr.aliyuncs.com`；配置文件权限设为 `0600`。
- `msmtp` 使用 `smtp.qq.com:465` TLS；邮箱授权码单独保存在 root 可读的 `0600` 文件中，由部署者在服务器隐藏输入，不通过聊天、工单或仓库传输。

QQ SMTP 授权码使用隐藏提示录入：

```bash
pwsh -NoLogo -NoProfile -File /opt/fengqigame/运维/配置备份告警凭据.ps1
```

将仓库部署到 `/opt/fengqigame` 后安装并启用定时器：

```bash
install -d -m 0700 /etc/fengqigame /opt/fengqigame/backups
install -m 0644 运维/systemd/fengqigame-backup.service /etc/systemd/system/
install -m 0644 运维/systemd/fengqigame-backup.timer /etc/systemd/system/
systemctl daemon-reload
systemctl enable --now fengqigame-backup.timer
systemctl start fengqigame-backup.service
systemctl status fengqigame-backup.service
systemctl list-timers fengqigame-backup.timer
```

timer 每天 `19:30 UTC` 运行，即北京时间 03:30，并启用 `Persistent=true`。人工邮件验收使用 `备份到OSS.ps1 -AlertTest`，不得通过破坏正式备份任务制造失败。

数据库和上传文件是两套独立恢复对象。建议用 Windows 任务计划程序每天分别执行：

```text
powershell.exe -NoProfile -ExecutionPolicy Bypass -File "F:\风起游戏管理后台\运维\备份数据.ps1"
powershell.exe -NoProfile -ExecutionPolicy Bypass -File "F:\风起游戏管理后台\运维\备份上传文件.ps1"
```

数据库恢复命令会要求输入数据库名确认，随后自动生成恢复前安全备份、停止应用写入、执行恢复，并在应用重新通过健康检查后才报告成功：

```powershell
.\运维\恢复数据.ps1 -BackupPath .\backups\fengqi-YYYYMMDD-HHMMSS.dump
```

上传文件恢复同样要求明确确认并自动生成安全快照；归档中的绝对路径或 `..` 路径会被拒绝：

```powershell
.\运维\恢复上传文件.ps1 -SnapshotPath .\backups\fengqi-uploads-YYYYMMDD-HHMMSS.tar.gz
```

2026-07-15 已完成正式验收：timer 实际运行、两类 OSS 对象 CRC64 与 AES-256 通过、删除/ACL/跨 Bucket/匿名访问被拒绝、告警负责人收到测试邮件，最新数据库与上传卷副本在独立 Compose 项目成功恢复并通过健康检查。数据库与上传文件仍应定期重复隔离恢复演练。

## 监控和排障

- 健康检查：`GET /api/system/health`
- 容器状态：`docker compose ps`
- 应用日志：`docker compose logs -f --tail 200 app`
- 数据库日志：`docker compose logs -f --tail 200 db`
- 反向代理日志：`docker compose logs -f --tail 200 caddy`
- 管理员页面“系统设置”显示账号、地图、有效会话和 24 小时审计事件数量。

登录失败时先确认浏览器访问地址与 `SITE_ADDRESS` 一致；HTTPS 域名必须设置 `COOKIE_SECURE=true`。本机 HTTP 调试才允许使用 `COOKIE_SECURE=false`。
