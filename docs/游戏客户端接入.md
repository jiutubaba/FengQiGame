# 游戏客户端接入

## 创建凭据

管理员或持有“管理游戏 API Key”权限的用户进入地图工作区“客户端接入”，为每个环境分别创建 Key。Token 只显示一次，后端只保存 SHA-256 哈希；遗失后应停用并重新创建。

请求统一携带：

```http
X-Map-Key: fqmap_生成的完整Token
Content-Type: application/json
```

Key 已绑定地图和环境，客户端请求体不要自行传 `mapId` 或环境。

## 接口与权限

| 权限                      | 接口                                                  | 用途                                |
| ------------------------- | ----------------------------------------------------- | ----------------------------------- |
| `game.players.write`      | `POST /api/game/players/upsert`                       | 按 UID 新增或更新玩家与最后活跃时间 |
| `game.logs.write`         | `POST /api/game/logs`                                 | 上报日志，相同内容自动聚合次数      |
| `game.metrics.write`      | `POST /api/game/metrics`                              | 按日期写入或更新地图指标            |
| `game.points.write`       | `POST /api/game/points/:pointKey/increment`           | 增加已启用埋点的触发次数            |
| `game.leaderboards.write` | `POST /api/game/leaderboards/:leaderboardKey/entries` | 批量新增或更新实时排行榜条目        |
| `game.risk.write`         | `POST /api/game/risk/events`                          | 按已启用规则幂等上报风险事件        |
| `game.messages.read`      | `GET /api/game/players/:uid/messages`                 | 拉取待送达消息                      |
| `game.messages.read`      | `POST /api/game/messages/:messageId/ack`              | 确认消息已经送达                    |
| `game.gifts.read`         | `GET /api/game/players/:uid/gifts`                    | 拉取待领取礼包                      |
| `game.gifts.read`         | `POST /api/game/gifts/:grantId/ack`                   | 确认礼包已经发入玩家存档            |

## 可靠领取流程

消息和礼包都采用“读取待处理记录 → 游戏内写入成功 → ACK”的顺序。只有游戏内写入成功后才能确认；未确认记录会在下一次拉取时再次返回，客户端因此必须按消息 ID 或发放 ID 实现幂等处理。

礼包 ACK 示例：

```json
{ "uid": "player-uid" }
```

指标上报示例：

```json
{
  "date": "2026-07-14",
  "cumulativeUsers": 1200,
  "onlineUsers": 35,
  "totalGameCount": 8190,
  "dailyNewUsers": 42,
  "dailyActiveUsers": 310,
  "lostUserCount": 70,
  "returnUserCount": 18,
  "activeUserRetentionRate": 31.5,
  "newUserRetentionRate": 22.4,
  "sevenDayRetentionRate": 14.8,
  "replayRate": 8.1
}
```

## 排行榜上报

后台先在当前地图和环境创建排行榜，并取得稳定的榜单 Key。客户端每次最多批量上报 500 条；相同 UID 会更新现有实时榜记录。被后台标记为“榜单封禁”的玩家不会进入实时排名和后续发布快照，已经发布的历史快照不会被改写。

```http
POST /api/game/leaderboards/game_power/entries
```

```json
{
  "entries": [
    {
      "uid": "player-001",
      "name": "玩家名称",
      "gameLevel": "N2",
      "score": 9900,
      "gameCount": 18,
      "metadata": { "season": 3 }
    }
  ]
}
```

## 风险事件上报

后台先创建并启用规则。`eventId` 是客户端生成的幂等键：网络重试必须继续使用同一个值，服务端不会重复创建事件。`details` 只能放处置所需的结构化证据，禁止上传账号口令、完整 Token 或其它无关敏感信息。

```http
POST /api/game/risk/events
```

```json
{
  "eventId": "server-01-20260714-000001",
  "ruleKey": "abnormal_power_growth",
  "uid": "player-001",
  "playerName": "玩家名称",
  "count": 3,
  "details": { "delta": 8800 },
  "occurredAt": "2026-07-14T20:30:00+08:00"
}
```

API 返回统一使用 `{ "success": true, "data": ... }`；失败时使用 `{ "success": false, "error": { "code", "message" }, "requestId" }`。客户端日志应记录 `requestId`，但不得记录完整 API Key。
