# 游戏客户端接入

需要直接复制到游戏项目的完整协议、字段约束、错误码、重试策略和 Lua 示例，见 [FQ 游戏服务器接口技术说明](FQ游戏服务器接口技术说明.md)。

## 创建凭据

管理员或持有“管理游戏 API Key”权限的用户进入地图工作区“客户端接入”，为每个环境分别创建 Key。Token 只显示一次，后端只保存 SHA-256 哈希；遗失后应停用并重新创建。

请求统一携带：

```http
FQ-Map-Key: fqmap_生成的完整Token
Content-Type: application/json
```

所有游戏端专属接口统一使用 `/api/fq` 前缀。Key 已绑定地图和环境，客户端请求体不要自行传 `mapId` 或环境。

| 权限                      | 接口                                                | 用途                                |
| ------------------------- | --------------------------------------------------- | ----------------------------------- |
| `game.archives.read`      | `POST /api/fq/bootstrap`                            | 批量读取玩家存档与全局存档          |
| `game.archives.read`      | `GET /api/fq/archives/players/:uid`                 | 读取单个玩家存档                    |
| `game.archives.read`      | `GET /api/fq/archives/global`                       | 读取当前地图环境的全局存档          |
| `game.archives.write`     | `POST /api/fq/archives/players/:uid/save`           | 带版本和幂等键保存完整玩家存档      |
| `game.archives.write`     | `POST /api/fq/archives/global/save`                 | 带版本和幂等键保存完整全局存档      |
| `game.players.write`      | `POST /api/fq/players/upsert`                       | 按 UID 新增或更新玩家与最后活跃时间 |

## 存档读取与保存

开局优先调用 `POST /api/fq/bootstrap`，一次读取本局涉及的玩家和地图全局存档：

```json
{
  "uids": ["player-001", "player-002"],
  "includeGlobal": true
}
```

返回的每份存档都有 `revision`。尚未保存过时返回 `revision: 0` 和空对象；被后台标记为“存档封禁”的玩家返回 `dataBanned: true`，不下发已有存档。

保存时提交完整存档对象，而不是数据库字段名或 SQL：

```http
POST /api/fq/archives/players/player-001/save
```

```json
{
  "requestId": "FQ-game-20260716-000001",
  "expectedRevision": 0,
  "values": {
    "gold": 100,
    "inventory": ["sword"],
    "progress": { "chapter": 2 }
  }
}
```

- `requestId` 必须以 `FQ-` 开头。同一次网络重试必须复用同一个 `requestId` 和完全相同的内容。
- `expectedRevision` 必须等于最近一次读取到的版本。保存成功后版本加一；版本不一致返回 `409 FQ_ARCHIVE_REVISION_CONFLICT`。
- 该存档没有发生后续更新前，相同 `requestId` 重放会返回保存结果并标记 `replayed: true`；同一 ID 改写内容返回 `409 FQ_REQUEST_REUSED`。更旧的迟到重试会被 revision 冲突安全拒绝。
- 玩家和全局存档单份上限为 512 KiB。JSON 整数如果可能超过 JavaScript 安全整数范围，应改用字符串。
- 游戏只能通过 FQ API 读写，不连接 PostgreSQL，也不在客户端保存数据库账号。

## 可靠领取流程

消息和礼包都采用“读取待处理记录 → 游戏内写入成功 → ACK”的顺序。只有游戏内写入成功后才能确认；未确认记录会在下一次拉取时再次返回，客户端因此必须按消息 ID 或发放 ID 实现幂等处理。

礼包 ACK 示例：

```json
{ "uid": "player-uid" }
```

指标上报示例：

```json
{
  "date": "2026-07-14",
  "cumulativeUsers": 1200,
  "onlineUsers": 35,
  "totalGameCount": 8190,
  "dailyNewUsers": 42,
  "dailyActiveUsers": 310,
  "lostUserCount": 70,
  "returnUserCount": 18,
  "activeUserRetentionRate": 31.5,
  "newUserRetentionRate": 22.4,
  "sevenDayRetentionRate": 14.8,
  "replayRate": 8.1
}
```

## 排行榜上报

后台先在当前地图和环境创建排行榜，并取得稳定的榜单 Key。客户端每次最多批量上报 500 条；相同 UID 会更新现有实时榜记录。被后台标记为“榜单封禁”的玩家不会进入实时排名和后续发布快照，已经发布的历史快照不会被改写。

```http
POST /api/fq/leaderboards/game_power/entries
```

```json
{
  "entries": [
    {
      "uid": "player-001",
      "name": "玩家名称",
      "gameLevel": "N2",
      "score": 9900,
      "gameCount": 18,
      "metadata": { "season": 3 }
    }
  ]
}
```

## 风险事件上报

后台先创建并启用规则。`eventId` 是客户端生成的幂等键：网络重试必须继续使用同一个值，服务端不会重复创建事件。`details` 只能放处置所需的结构化证据，禁止上传账号口令、完整 Token 或其它无关敏感信息。

```http
POST /api/fq/risk/events
```

```json
{
  "eventId": "server-01-20260714-000001",
  "ruleKey": "abnormal_power_growth",
  "uid": "player-001",
  "playerName": "玩家名称",
  "count": 3,
  "details": { "delta": 8800 },
  "occurredAt": "2026-07-14T20:30:00+08:00"
}
```

API 返回统一使用 `{ "success": true, "data": ... }`；失败时使用 `{ "success": false, "error": { "code", "message" }, "requestId" }`。客户端日志应记录 `requestId`，但不得记录完整 API Key。

## 《沧澜》现有框架如何读取

当前《沧澜》已完成 FQ 适配层实现，不再连接旧 `/war3/...` 接口。服务器框架入口位于 `scripts/maps/server/init.lua`，由 `scripts/maps/init.lua` 用“风起服务器加载完毕 / 加载失败”门闩控制游戏模块启动；核心实现位于 `FQHttpClient.lua` 与 `FQServer.lua`。

`FQHttpClient` 固定请求 `https://fengqigame.com/api/fq/**`，并统一传入：

```lua
headers = {
    ['FQ-Map-Key'] = FQ_API_KEY,
}
```

当前读取顺序：

1. 先沿用现有平台逻辑取得本局玩家 UID。
2. 由一个本地客户端调用 `POST /api/fq/bootstrap`，`uids` 传本局全部玩家 UID。
3. `JSON.decode(resp.body)` 后判断 `result.success == true`，不再判断旧格式的 `result.status == 'success'`。
4. 将 `result.data.global.values` 同步进 `ServerConfig.Globals`。
5. 按 `result.data.players` 中的 `uid` 找到玩家句柄，把对应 `values` 放入 `ServerConfig.PlayerData`；同时保存每个玩家的 `revision`，后续保存时作为 `expectedRevision`。
6. 保存成功后用响应中的新 `archive.revision` 覆盖本地版本；遇到 `FQ_ARCHIVE_REVISION_CONFLICT` 时重新读取，不能盲目重试覆盖。

当前实现已经选择原始 JSON：`ServerPlayerData` 保留字符串、数字、布尔与表，不再对 FQ 值执行旧 Base64 解码。玩家和全局完整快照分别维护 revision，保存重试复用完全相同的 requestId、expectedRevision 与请求正文。

在当前原始 JSON 实现下，数据库表名、数据库密码和 SQL 都不会出现在游戏代码中。数据库维护由后台和运维脚本负责，地图只认识 FQ API、UID、JSON 存档、revision 与 requestId。

代码实现不等于真实环境已经验收：本机 `scripts/maps/server/FQPrivateConfig.lua` 仍未创建，三套正式 Key 当前还缺少消息/礼包读取权限。完整交接状态、权限缺口与实机顺序见 [沧澜项目 AI 技术交接](沧澜项目AI技术交接.md)。
